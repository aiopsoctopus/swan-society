01 — The Problem

Book Clubs Die From the Inside

The person who starts the book club is rarely the problem. The reading, the wine, the conversation — people love all of that. What kills a club is the invisible overhead that falls on one person, month after month: picking a book everyone will like, herding RSVPs, generating discussion questions, making sure the next host is ready to go.

"No one quits book club because they stopped liking books. They quit because being the host stopped being worth it."

The Trigger
Host Burnout
Coordination overhead — RSVPs, book selection politics, writing discussion questions from scratch — lands entirely on one person. Over time, the role stops feeling like a gift and starts feeling like a second job.
The Gap
No Tool Owns This
Fable was acquired by Everand in June 2026 and is pivoting to content subscriptions. Bookclubs.com has broad features but isn't built around the host's specific burnout problem. The host-operations gap is real and widening.
The Wedge
AI at Two Touchpoints
Book selection (narrowing a genre-matched shortlist) and discussion questions (generating 8 thoughtful prompts instantly) are the two highest-effort host tasks. Both are now solvable with one Groq API call each.
The Model
Host Pro · $5/mo
Free tier establishes the habit. Pro unlocks the AI features (discussion questions, theme suggestions) and host-rotation automation — the features that make the tool worth keeping month after month.

02 — Why Now

Three Forces Converging

Book clubs are growing. Gen Z and millennials are forming genre-specific clubs at breweries, on group runs, at dating events. BookTok — TikTok's most popular community — is normalizing reading as identity and converting that energy into real-world gatherings. The 31% event growth isn't a blip.

The incumbent just left the space. Fable's acquisition by Everand (June 2026) shifts its focus to content and subscriptions, not host operations. The gap is real and opening wider.

AI makes the hard part easy. Generating 8 thoughtful, book-specific discussion questions used to require research or a lot of thought. Now it's one API call. The product category is newly possible in a way it wasn't 18 months ago.

04 — Build Plan

48 Hours · 15 Prompts

The entire app is built through 15 structured prompts sent to Lovable via the Claude connector. Each prompt has a clear scope, a verification gate, and a dependency on the one before it. The goal: no ambiguity mid-build, no 2am architectural decisions.

"A smaller app that works end to end is a stronger submission than a bigger app with one visible broken state."

Hour-by-Hour Schedule
Hr 1–2 · Day 1 Morning
Foundation Must-have
Create project → design system + auth → Supabase schema (all 6 tables) → Row-Level Security. Nothing gets built until auth and the database are confirmed working.
Hr 2–3 · Day 1 Morning
Club Creation + Dashboard Must-have
First-run welcome screen writes a clubs row to Supabase. Dashboard reads from it on mount. Manual gate: confirm the data round-trip works before building anything on top of it.
Hr 3–5 · Day 1 Midday
AI Book Suggestions + Voting Must-have
The spine of the app. Client owns pool state (12 hardcoded 2026 releases, 2 per genre). Edge function writes descriptions only — never selects or invents titles. Votes persist in Supabase with realtime tallies.
Hr 6–8 · Day 1 Afternoon
RSVP Screen Must-have
Upserts to rsvps table on every click. Live avatar list via Supabase realtime. Empty state if no meetup is scheduled — no broken date fields.
Hr 9–11 · Day 1 Evening
AI Discussion Questions Differentiator
The "wow" feature, built while sharp — not at hour 40. Live Groq call (Llama 3.3 70B) with 8s timeout and hardcoded fallback sets for three demo books. Regenerate is the primary action; Save as PDF is de-emphasized.
Hr 14–17 · Day 2 Morning
Monetization UI Must-have
Pricing screen with Free + Pro tiers. Upgrade CTA opens a modal with waitlist email capture — saves to Supabase. No dead ends, no placeholder buttons.
Hr 22–25 · Day 2 Afternoon
Polish Pass Must-have
Empty states, mobile responsiveness (≤768px), visual consistency. This is where "beautiful" actually gets judged.
Hr 26–28 · Day 2 Evening
Deploy + Cold Judge Test Must-have
Deploy to live URL. Open fresh on a different device and network. Walk the full path end to end as an unguided judge would. Fix anything broken. This block exists so Day 3 isn't a fire drill.
Hr 18–21 · Day 2 Midday (if ahead)
Nice-to-Haves Optional
Affiliate links, host package screen, rotation view. Skip all three and go straight to polish if behind. A polished core loop beats a half-built feature set every time.

Build Philosophy
01
Schema first, screens second
All 6 tables are created before any screen prompt. Without this, Lovable generates inconsistent column names across prompts and the joins between screens don't work.
02
Client owns pool state
The book pool JSON is embedded in the frontend. Swap is a client-side array lookup. The edge function writes descriptions only — never selects or invents titles. Hallucinated books are impossible.
03
Fallbacks before features
Every AI touchpoint has a hardcoded fallback. Groq fails → cached questions appear. No spinners that never resolve. No blank screens. Judges don't get narration.
04
Pre-decided triage rules
Five scenarios are mapped out in advance: what to cut, when to cut it, what never gets cut. These decisions are made now so they don't get made at hour 38 on two hours of sleep.
